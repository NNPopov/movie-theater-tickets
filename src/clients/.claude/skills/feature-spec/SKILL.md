# Как писать Feature Spec для нового слайса

Этот документ — инструкция по созданию промпта для реализации нового слайса.
Основан на реальном примере `delete_user` слайса из фичи `users`.

---

## Структура промпта

Промпт состоит из 7 блоков. Каждый блок обязателен.

```
1. ЗАГОЛОВОК     — одна строка: что делаем и зачем
2. КОНТЕКСТ      — что читать, что не читать
3. API           — эндпоинт, параметры, ответы, ошибки
4. СТРУКТУРА     — целевое дерево папок и файлов
5. ЧТО ДЕЛАТЬ    — пошаговая реализация по слоям
6. ТЕСТЫ         — что покрывать в каждом файле
7. ОТЧЁТ         — что агент должен предоставить по завершении
8. ЧТО НЕ ДЕЛАТЬ — явные ограничения scope
```

---

## Блок 1: ЗАГОЛОВОК

Одна-две строки. Отвечает на вопросы: что создаём, в какой фиче, какой UX-результат.

**Шаблон:**
```
Задача: создать новый слайс {slice_name} в фиче {feature_name}.
{Краткое описание бизнес-логики и UX-результата}.
```

**Пример (delete_user):**
```
Задача: создать новый слайс delete_user в фиче users.
Удаление своего аккаунта с правильным UX (диалог подтверждения,
локальный logout после успеха, редирект на публичный /users со снекбаром).
```

**Правила:**
- Не описывай технические детали — это для блока "Что делать"
- Упомяни UX-поток если он нетривиальный (диалог, редирект, snackbar)
- Если слайс не имеет своего экрана — скажи об этом явно

---

## Блок 2: КОНТЕКСТ

Самый важный блок. Агент читает только то что ты указал.

**Шаблон:**
```
====== КОНТЕКСТ ======

ЧИТАЙ:
- @CLAUDE.md полностью
- @{аналогичный_слайс}/** — как ближайший аналог (НЕ копировать, опираться)
- @{файлы_которые_будут_изменены}
- @{общие_файлы_фичи} — _shared/
- @{core_зависимости}
- @{нужные_skills}

НЕ ЧИТАЙ:
- @{несвязанные_слайсы}/**
- @{тяжёлые_файлы_без_нужды}
```

**Что всегда включать в ЧИТАЙ:**
- `@CLAUDE.md` — всегда, без исключений
- Ближайший аналог-слайс с пометкой "НЕ копировать, опираться"
- Файлы которые будут **изменены** (не только созданы)
- `_shared/` файлы фичи: api_client, entities
- core зависимости: auth, errors, routing — только те что реально нужны
- Релевантные skills из `.claude/skills/`

**Что включать в НЕ ЧИТАЙ:**
- Другие слайсы той же фичи если они не нужны
- Тяжёлые generated файлы (*.gr.dart, *.config.dart)
- Слайсы других фич

**Пример (delete_user):**
```
ЧИТАЙ:
- @CLAUDE.md полностью
- @lib/core/auth/application/auth_cubit.dart — для forceLogout
- @lib/core/auth/infrastructure/token_holder.dart
- @lib/features/users/_shared/data/users_api_client.dart
- @lib/features/users/_shared/domain/entities/user.dart
- @lib/features/users/user_details/presentation/user_details_screen.dart
  — туда добавим кнопку Delete рядом с Edit
- @lib/features/users/edit_user/** — как ближайший аналог слайса
- @lib/core/errors/failure.dart
- @lib/core/routing/app_router.dart
- @.claude/skills/bloc/SKILL.md

НЕ ЧИТАЙ:
- @lib/features/users/list_users/application/**, data/**, domain/**
- @lib/features/users/create_user/**
- @lib/features/auth/login/**
```

## Блок 3: API

Полный контракт с бэкендом. Агент не должен ничего угадывать.

**Шаблон:**
```
====== API ======

{HTTP_МЕТОД} {полный_url_с_параметрами}
Header: Authorization: Bearer <token>  (если нужен)
Body: {если есть}
Ответ 200: {точный JSON}

Важные особенности:
- {нетривиальное поведение сервера}
- {side effects на стороне сервера}

Ошибки:
- {код} — {что означает}
- {код} — {что означает}
```

**Правила:**
- Указывай **все** коды ошибок которые сервер может вернуть
- Описывай side effects сервера (например: токен попадает в blacklist)
- Если поведение сервера влияет на клиентский flow — объясни последствие
- Указывай тип возвращаемого значения (`Future<void>` vs `Future<T>`) и почему

**Пример (delete_user):**
```
====== API ======

DELETE http://127.0.0.1:8000/api/v1/user/{username}
Header: Authorization: Bearer <token>
Ответ 200: {"message": "User deleted"}

Важные особенности:
- Сервер разрешает удалять ТОЛЬКО свой профиль (403 иначе)
- После успешного DELETE сервер ЗАНОСИТ ТОКЕН В BLACKLIST
- Значит следующий запрос по старому токену вернёт 401
- Задача: сделать явный локальный logout ДО того, как interceptor
  наткнётся на 401 от случайного фонового запроса

Ошибки:
- 401 — токен невалиден
- 403 — не свой профиль
- 404 — юзер не найден
```

## Блок 4: СТРУКТУРА

Целевое дерево файлов. Агент создаёт ровно это — не больше, не меньше.

**Шаблон:**
```
====== Целевая структура ======

lib/features/{feature}/{slice}/
├── domain/
│   ├── ports/
│   │   └── {slice}_port.dart        # сигнатура метода
│   └── usecases/
│       └── {slice}_usecase.dart
├── data/
│   └── {slice}_adapter.dart
├── application/
│   ├── {slice}_cubit.dart
│   └── {slice}_state.dart
└── presentation/
    ├── {slice}_screen.dart           # если есть экран
    ├── {slice}_route.dart            # если есть маршрут
    └── widgets/
        └── {widget}.dart
```

**Правила:**
- Комментируй нетривиальные файлы прямо в дереве
- Если слайс БЕЗ экрана — укажи это явно и объясни почему
- Если добавляется метод в _shared — укажи это отдельно

**Пример (delete_user — слайс без экрана):**
```
====== Целевая структура ======

lib/features/users/delete_user/
├── domain/
│   ├── ports/
│   │   └── delete_user_port.dart    # Future<Either<Failure, Unit>> call(String username)
│   └── usecases/
│       └── delete_user_usecase.dart
├── data/
│   └── delete_user_adapter.dart
├── application/
│   ├── delete_user_cubit.dart
│   └── delete_user_state.dart       # sealed через freezed
└── presentation/
    ├── delete_account_button.dart   # виджет-кнопка с диалогом
    └── widgets/
        └── delete_confirmation_dialog.dart

Особенность: у delete_user НЕТ своего экрана и маршрута.
Presentation/ содержит только виджеты, встраиваемые в user_details_screen.
```

## Блок 5: ЧТО ДЕЛАТЬ

Пошаговая реализация. Нумерованные шаги по слоям архитектуры.

**Порядок шагов всегда:**
```
1) _SHARED (если нужны изменения в общем API клиенте)
2) DOMAIN  (port → usecase)
3) DATA    (adapter)
4) APPLICATION (state → cubit)
5) PRESENTATION (экран/виджеты)
6) ИНТЕГРАЦИЯ (изменения в существующих файлах)
7) ЛОКАЛИЗАЦИЯ (ключи для slang)
```

**Уровень детализации по ситуации:**

| Ситуация | Детализация |
|---|---|
| Стандартный слайс | Только сигнатуры + нетривиальные части |
| Нестандартный flow (например, forceLogout) | Полный код метода с объяснением |
| Паттерн уже есть в аналоге | "Смотри edit_user, делай аналогично" |
| Критическая ошибка возможна | Полный код + объяснение почему именно так |

**Для каждого шага указывай:**
- Файл(ы) которые создаются или изменяются
- Что именно делать (сигнатуры, логика, ограничения)
- Почему если решение нетривиальное

**Пример нетривиального пояснения (из delete_user):**
```
КРИТИЧЕСКОЕ: forceLogout принимает параметр notifyUser: false, чтобы
НЕ показывался глобальный snackbar "Session expired". Мы показываем
СВОЙ snackbar "Account deleted" в UI.

Если текущая сигнатура forceLogout не принимает notifyUser — добавь
параметр со значением по умолчанию true, чтобы не ломать существующие вызовы.
```

## Блок 6: ТЕСТЫ

Всегда в конце блока "Что делать", отдельным пунктом.

**Шаблон:**
```
N) ТЕСТЫ

   test/features/{feature}/{slice}/:

   a) application/{slice}_cubit_test.dart:
      - {сценарий} → emits [{состояния}]
      - {сценарий} → emits [{состояния}], проверить что {mock}.{метод} вызван
      - {негативный сценарий} → emits [{состояния}], {mock}.{метод} НЕ вызван

   b) data/{slice}_adapter_test.dart:
      - success: api вернул OK → Right(...)
      - {код ошибки} → Left({FailureType})
      - unexpected {Exception} → Left(UnknownFailure), logger.error вызван
```

**Правила:**
- Для каждого Cubit: happy path + все failure пути + cancel/dismiss если есть
- Для каждого Adapter: success + каждый HTTP код ошибки + unexpected exception
- Всегда проверяй что logger.error вызван при unexpected exception
- Всегда проверяй что деструктивные side effects (logout, redirect) вызваны/не вызваны

---

## Блок 7: ОТЧЁТ

Что агент должен предоставить по завершении.

**Стандартный шаблон:**
```
====== ОТЧЁТ ======

- Список файлов (новых + изменённых)
- Подтверждение что другие слайсы НЕ трогались
- Список изменений в core/ если были
- UX-прогон: все шаги проверены
- Тесты: сколько новых, все зелёные
```

---

## Блок 8: ЧТО НЕ ДЕЛАТЬ

Явные ограничения. Предотвращает типичные ошибки агента.

**Что всегда включать:**
- Scope ограничения (какие файлы НЕ трогать)
- Архитектурные запреты специфичные для слайса
- Анти-паттерны которые агент может предложить "для удобства"

**Пример (delete_user):**
```
====== ЧТО НЕ ДЕЛАТЬ ======

- НЕ создавать отдельный маршрут delete_user_route — delete не имеет экрана
- НЕ вызывать регулярный logout() через API (сервер вернёт 401)
- НЕ показывать snackbar "Session expired" — у нас свой "Account deleted"
- НЕ удалять токен из storage до API-вызова — если запрос упадёт,
  юзер должен остаться залогиненным
- НЕ добавлять кнопку никуда кроме user_details_screen
```

## Чеклист перед отправкой промпта

```
□ Заголовок объясняет бизнес-результат, не технические детали
□ ЧИТАЙ включает CLAUDE.md
□ ЧИТАЙ включает аналогичный слайс с пометкой "опираться, не копировать"
□ ЧИТАЙ включает все файлы которые будут изменены
□ НЕ ЧИТАЙ исключает несвязанные слайсы
□ API описывает все коды ошибок
□ API описывает все side effects на стороне сервера
□ Структура файлов полная и с комментариями
□ Нетривиальные решения объяснены с пометкой КРИТИЧЕСКОЕ или ВАЖНО
□ Тесты покрывают happy path + все failure + side effects
□ ЧТО НЕ ДЕЛАТЬ закрывает очевидные соблазны для агента
□ ОТЧЁТ запрашивает подтверждение что чужие слайсы не тронуты
```

---

## Когда детализировать больше, когда меньше

**Больше деталей если:**
- Нетривиальный flow (logout после delete, blacklist токена)
- Изменяется core/ (auth, routing, DI)
- Слайс без экрана (нестандартная структура)
- Критические side effects которые нельзя напутать

**Меньше деталей если:**
- Слайс аналогичен существующему (достаточно "смотри edit_user")
- Стандартный CRUD без особенностей
- Агент уже делал похожий слайс в этой сессии

**Золотое правило из SDD курса:**
> Много контекста о целях и ограничениях.
> Мало указаний о низкоуровневых решениях которые агент решит сам.
