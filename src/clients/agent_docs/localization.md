# Localization reference

Read this when adding or modifying any UI string, supporting a new locale, or working
with `LocaleCubit`.

## How localization is wired

- String sources are JSON files under `lib/core/i18n/i18n/` —
  `strings.en.json`, `strings.ru.json`, and so on.
- Codegen tool: `dart run slang`. Run after **every** edit to a JSON source.
- In code, strings are accessed type-safely:
  `context.t.auth.login.title`.
- The active locale is controlled by `LocaleCubit` in `core/i18n/`. UI never reads the
  device locale directly.

## Key organization

Keys are organized **by feature**, mirroring the folder structure of `lib/features/`.
Top-level keys are feature names; nested keys are slices and elements inside the slice.

```json
{
  "auth": {
    "login": {
      "title": "Sign in",
      "emailHint": "Email"
    }
  }
}
```

Cross-cutting strings (errors, generic actions like "Cancel"/"OK", validation messages)
live under top-level keys `common`, `errors`, and `validation`.

## Hard rules

- ❌ Hardcoding any UI string. Even a one-character button label goes through `slang`.
- ❌ Concatenating localized strings: `${t.x.y} ${t.x.z}`. Use parameterized keys
  instead — `slang` supports `{name}` placeholders.
- ❌ Using `Intl.message` or `flutter_localizations` directly. The project uses `slang`
  exclusively.
- ❌ Editing a `*.g.dart` file generated by `slang`. Edit the JSON, then re-run codegen.
- ❌ Adding a key to one locale but not the others. `slang` will warn at codegen time;
  treat that warning as an error.

## When a string is a temporary placeholder

If a quick prototype is needed and the final wording is not yet decided, use the key
`wip.todoReplace` with a meaningful working value, like
`"todoReplace": "TODO: rename this confirm button"`. Searching the codebase for
`wip.todoReplace` then surfaces every placeholder before release.

## Adding a new locale

1. Create `lib/core/i18n/i18n/strings.<lang>.json` with the same key shape as
   `strings.en.json`.
2. Run `dart run slang`.
3. Register the locale in `LocaleCubit` (the list of supported locales).
4. Verify the locale switcher in settings shows the new option.

## Common mistakes

- ❌ Adding a string only to the English JSON and shipping. The other locales fall back
  silently and the bug shows up in production.
- ❌ Putting a feature-specific string under `common.*`. If only one feature uses it,
  it belongs under that feature's key.
- ❌ Forgetting to re-run `dart run slang` after a JSON change. The compiler will not
  catch the missing key until the generated file is updated.
